# 日报技能意图路由表

本文件用于把用户自然语言请求先路由成单一动作，再进入固定执行流程，降低模型在第一步就选错工具的概率。

## 路由原则

- 先判断用户要“查”“列”“提醒”哪一类动作
- 优先路由到单个工具；只有提醒未交日报这类场景才进入组合流程
- 同一句话如果同时包含“查询 + 提醒”，先查再提醒
- 缺少关键参数时，只追问缺失部分，不改动作类型

## 一级路由

| 用户意图 | 触发词示例 | 路由结果 |
| ---- | ---- | ---- |
| 查询日报 | 日报、工作总结、今日进展、本周汇报、今天写了什么 | `woa_daily` |
| 查询成员 | 日报人员、谁需要写日报、谁要写日报、日报名单 | `woa_daily_members` |
| 提醒未交日报 | 提醒日报、催日报、提醒没写日报的人、谁没写日报并提醒 | 组合流程 |

## 二级路由

### A. 路由到 `woa_daily`

适用话术：

- 查今天日报
- 看张三本周日报
- 李四上周工作总结
- 这个月谁写了日报

参数映射：

- 时间范围 -> `period`
- 指定姓名 -> `person`

进入流程：

- 走 `查询脚本`
- 参考 `query-examples.md`

### B. 路由到 `woa_daily_members`

适用话术：

- 日报人员有哪些
- 谁需要写日报
- 拉一下日报名单

进入流程：

- 走 `成员脚本`

### C. 路由到组合流程

适用话术：

- 提醒没写日报的人
- 催一下今天没交日报的人
- 看看谁没写日报并提醒一下

进入流程：

1. `woa_daily_members`
2. `woa_daily` with `period=today`
3. 求差集
4. `woa_daily_remind`

进入参考：

- 参考 `remind-workflow.md`

## 路由判定细则

### 1. 同时出现“谁没写日报”和“提醒”

判定为提醒流程，不要只做查询。

### 2. 同时出现“日报人员”和“提醒”

如果用户是在问名单，先路由到 `woa_daily_members`。
如果用户明确要求提醒，路由到组合流程。

### 3. 只出现“日报”但没有时间范围

动作仍然是 `woa_daily`，但先追问 `period`。

### 4. 只出现人名，没有时间范围

动作仍然是 `woa_daily`，保留 `person`，只追问 `period`。

### 5. 指定提醒某个人

动作类型仍然是提醒，但提醒前必须通过成员列表确认有效 `member_id`。

## 推荐顺序

处理相关请求时，建议按下面顺序思考：

1. 这是查询、列成员，还是提醒
2. 是否需要组合流程
3. 是否缺少 `period` 或目标成员
4. 进入对应参考文档执行

## 不推荐做法

- 不要先查接口细节再决定意图
- 不要把“提醒谁没写日报”误判成普通查询
- 不要把“日报人员”误判成日报内容查询
- 不要在第一步同时发起多个不必要的工具调用
