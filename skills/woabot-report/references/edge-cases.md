# 日报技能边界场景

本文件用于约束容易出现不稳定调用的边界情况。

## 1. 用户没说时间范围

场景：

- 查日报
- 看一下日报
- 张三写日报了吗

处理规则：

- 如果上下文已有明确时间范围，直接继承
- 如果上下文没有明确时间范围，只追问一次
- 不要擅自默认成 `today`，除非用户表达就是“今天”语境

推荐追问：

```text
你要查今天、本周、上周、本月还是上月的日报？
```

## 2. 用户只说人名，不说时间

场景：

- 查张三日报
- 李四有发日报吗

处理规则：

- 保留 `person`
- 只补问 `period`

## 3. 人名模糊或多人重名

场景：

- 查小张日报
- 提醒王工写日报

处理规则：

- 优先按已有成员列表做匹配
- 如果可能匹配到多人，先澄清，不要猜
- 不要在未确认的情况下直接提醒

推荐追问：

```text
匹配到多个成员，请确认你要查询或提醒的是哪一位。
```

## 4. 没查到日报

处理规则：

- 直接说明未查到
- 不要暗示接口异常，除非接口确实报错
- 不要自动改查其他时间范围

推荐回复：

```text
未查询到{时间范围}{姓名可选}的日报。
```

## 5. 没有可提醒成员

处理规则：

- 不调用 `woa_daily_remind`
- 直接说明无需提醒或没有有效成员

推荐回复：

```text
今天所有应写成员都已提交日报，无需提醒。
```

## 6. 提醒接口部分失败

处理规则：

- 保留成功结果
- 同时说明跳过和失败项
- 不要把整体结果笼统说成“失败”

推荐回复：

```text
已发送日报提醒。
成功：{sent}
跳过：{skipped}
失败：{errors}
```

## 7. 用户要求提醒某个人

场景：

- 提醒张三写日报
- 催李四交日报

处理规则：

- 先通过成员列表确认 `member_id`
- 找不到有效成员时，不发送提醒
- `message` 中不要拼接人名和 `@`

## 8. 用户只想看结果，不想看过程

处理规则：

- 返回整理后的结论
- 不输出原始请求参数和接口结构
- 除非用户明确要求，否则不展示 `member_id`、`chat_id` 之类底层字段
